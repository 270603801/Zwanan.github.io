    结构型模式

    发表于 2022-12-03 更新于 2023-03-01 分类于 学习
    设计模式---结构型模式

    1.装饰者模式(包装模式)

    装饰者模式(Decorator Pattern)：动态地给一个对象添加一些额外的职责。就增加功能来说，装饰者模式相比生成子类更为灵活。

    装饰者模式是一种对象结构型模式。
    装饰者模式包含如下角色：

    1. 抽象构件(Component)角色：给出一个抽象接口，以规范准备接收附加责任的对象。
    2. 具体构件(ConcreteComponent)角色：定义一个将要接收附加责任的类。
    3. 装饰(Decorator)角色：持有一个构件(Component)对象的实例，并定义一个与抽象构件接口一致的接口。
    4. 具体装饰(ConcreteDecorator)角色：负责给构件对象“贴上”附加的责任。
    5. 客户端(Client)角色：装饰模式的客户端充当了使用一个具体构件对象和装饰对象之间的粘合剂的角色，客户端并不知道具体构件对象所具有的功能，只有在运行时才能够确定具体构件对象的具体功能。
    优点
    • 装饰者模式与继承关系的目的都是要扩展对象的功能，但是装饰者模式可以提供比继承更多的灵活性。
    • 可以通过一种动态的方式来扩展一个对象的功能，通过配置文件可以在运行时选择不同的装饰器，从而实现不同的行为。
    • 可以对一个对象进行多次装饰，创造出不同行为的组合，得到功能更为强大的对象。
    缺点
    • 使用装饰者模式进行系统设计时将产生很多小对象，如果过度使用会使系统变得很复杂。
    适用场景
    • 在不影响其他对象的情况下，以动态、透明的方式给单个对象添加职责。
    • 处理那些可以撤消的职责。
    • 当不能采用继承的方法进行扩充时。需要增加太多基本功能的排列组合时，采用继承关系很难实现，而采用装饰者模式却很好实现。
    类图

    打印账单的例子，打印账单的时候，可能需要打印账单的头部，也可能需要打印账单的尾部，也可能需要打印账单的中间部分，这些都是可选的，所以我们可以使用装饰者模式来实现。

    Bill为抽象构件，Concrete为具体构件，BillDecorator为抽象装饰者，HeaderBill、FootBill为具体装饰者。

    2.代理模式

    代理模式是一种对象结构型模式。
    意图：为其他对象提供一种代理以控制对这个对象的访问。
    代理模式包含以下主要角色。

    1. 抽象主题(Subject)角色：声明真实主题和代理主题的共同接口，这样一来在任何可以使用真实主题的地方都可以使用代理主题，客户端通常需要针对抽象主题角色进行编程。
    2. 真实主题(RealSubject)角色：定义代理角色所代表的真实对象，是我们最终要引用的对象。
    3. 代理(Proxy)角色：保存一个引用使得代理可以访问实体，并提供一个与Subject的接口相同的接口，这样代理就可以用来代替实体。
    4. 客户(Client)角色：通过一个代理访问一个实体，因此要使用代理的功能。
    优点
    • 代理模式能够协调调用者和被调用者，在一定程度上降低了系统的耦合度。
    缺点
    • 代理模式会造成系统设计中类的数目增加。
    • 在客户端和目标对象之间增加一个代理对象，会造成请求处理速度变慢。
    • 需要事先确定代理类的性质
    适用场景
    • 远程代理：为一个对象在不同的地址空间提供局部代表。这样可以隐藏一个对象存在于不同地址空间的事实。
    • 虚拟代理：根据需要创建开销很大的对象。通过它来存放实例化需要很长时间的真实对象。
    • 安全代理：用来控制真实对象访问时的权限。例如，网上商城中对不同的用户提供不同的访问权限。
    • 智能指引：当调用真实的对象时，代理处理另外一些事。
    类图

    使用拨电话和接电话的例子来说明代理模式的结构和特点。
    Listener就是抽象类，ProxyListener就是代理类，RealListener就是真实类。

    代理类和真实类都实现了抽象类的接口，代理类中持有真实类的引用，代理类中的方法调用真实类的方法，代理类中可以增加一些额外的操作，比如打印日志等。

    3.适配器模式

    适配器模式(Adapter Pattern)：将一个类的接口转换成客户希望的另外一个接口。适配器模式使得原本由于接口不兼容而不能一起工作的那些类可以一起工作。

    适配器模式分为类适配器模式和对象适配器模式，类适配器模式使用多重继承，对象适配器模式使用组合。

    专门用在两个接口不兼容的情况下，比如两个接口的方法名不一样，或者两个接口的参数不一样，或者两个接口的返回值不一样，这时候就需要适配器模式来适配两个接口。
    还有系统后期扩展，需要使用新的接口，但是又不想改变原来的代码，这时候就可以使用适配器模式来适配新的接口。

    主要包含的角色：
    1.目标接口：客户所期待的接口，目标可以是具体的或抽象的类，也可以是接口。
    2.源角色：需要适配的类或适配者类。
    3.适配器：通过包装一个源对象，把源对象的接口转换成目标接口。

    优点
    • 可以让任何两个没有关联的类一起运行。
    • 提高了类的复用。
    • 增加了类的透明度。
    • 灵活性好。
    缺点
    • 过多的使用适配器，会让系统非常凌乱，不易整体进行把握。
    • 类适配器不能适配一个类和它的子类。
    适用场景
    • 系统需要使用现有的类，而此类的接口不符合系统的需要。
    • 当创建一个可以复用的类，用于与其他不相关的类或不可预见的类协同工作时。
    • 在对象适配器中要匹配多个子类的情况，可以使用类适配器，为对象适配器可以适配他们父类的接口。
    类图

    使用一个适配原来保存日志的接口，但是现在需要保存到数据库中，这时候就需要适配器模式来适配新的接口。来适配新的操作
    LogFileOperator是被适配的接口，LogDBOperator是适配后的新接口，Adapter是适配器，LogModel是日志信息的实体类

    4.外观模式

    外观模式(Facade Pattern)：为子系统中的一组接口提供一个一致的界面，此模式定义了一个高层接口，这个接口使得这一子系统更加容易使用。

    外观模式的主要角色如下：

    1.外观(Facade)角色：为多个子系统对外提供一个共同的接口。

    2.子系统(Sub System)角色：实现系统的部分功能，客户可以通过外观角色访问它。

    优点
    • 减少系统相互依赖。
    • 提高灵活性。
    • 提高了安全性。
    缺点
    • 如果设计不当，增加新的子系统可能需要修改外观类或客户端的源代码，违背了“开闭原则”。
    • 不能很好地限制客户使用子系统类，如果对客户访问子系统类做太多的限制则减少了可变性和灵活性。
    适用场景
    • 对于一个复杂的子系统，需要为用户提供一个简单的交互操作。
    • 客户端与多个子系统之间存在很大的依赖性。引入外观类将子系统与客户端解耦，可以提高子系统的独立性和可移植性。
    • 当整个系统需要构建一个层次结构的子系统，不希望子系统直接的交互。
    类图

    其中SubSystem类都是子系统，ReadAndWriteFacade类是客户端，它需要调用子系统的方法，但是不需要知道子系统的具体实现，只需要知道子系统的接口就可以了，这样就实现了客户端和子系统的解耦。

    5.组合模式(合成模式)

    组合模式(合成模式), 也称为部分-整体模式，主要是用来描述部分与整体的关系。使得用户对单个对象和组合对象的使用具有一致性。

    组合模式的主要角色如下：

    1.抽象构件(Component)角色：它是组合中对象声明接口，在适当的情况下，实现所有类共有接口的缺省行为。声明一个接口用于访问和管理Component的子部件。

    2.叶子构件(Leaf)角色：在组合中表示叶子节点对象，叶子节点没有子节点。

    3.树枝构件(Composite)角色：它的作用是存储子部件，在Component接口中实现与子部件有关的操作，如增加(add)和删除(remove)等。

    优点
    • 定义了包含对象和组合对象的类层次结构，使得客户端可以一致地使用组合结构和单个对象，高层模块调用的更加简单。
    • 客户端不必关心处理的是单个对象还是组合对象，简化了客户端代码。
    • 更容易在组合体内加入对象构件，客户端不必因为加入了新的对象构件而更改源代码，符合“开闭原则”。
    缺点
    • 使设计变得更加抽象，对象的业务规则如果很复杂，则实现组合模式具有很大挑战性，而且不是所有的方法都与叶子对象子类都有关联。
    • 限制了接口的影响范围，使得设计更加有限制性，违背了依赖倒置原则（通过抽象（接口、抽象类）使各个类或模块的实现彼此独立，不互相影响，实现模块间的松耦合）。
    类图

    其中Component是抽象构件，Leaf是叶子构件，Composite是树枝构件，Client是客户端，它需要调用Component的方法，但是不需要知道Component的具体实现，只需要知道Component的接口就可以了，这样就实现了客户端和Component的解耦。

    6.桥接模式

    桥接模式(Bridge)是一种结构型设计模式，它将抽象部分与它的实现部分分离，使它们都可以独立地变化。桥接模式是一种对象结构型模式，又称为柄体(Handle and Body)模式或接口(Interface)模式。

    桥接模式的主要角色如下。

    1.Abstraction（抽象类）：定义抽象类的接口，它一般是抽象类而不是接口，其中定义了一个Implementor（实现类接口）类型的对象并可以维护该对象，它与Implementor之间具有关联关系，它既可以包含抽象业务方法，也可以包含具体业务方法。

    2.RefinedAbstraction（扩充抽象类）：扩充由Abstraction定义的接口，通常情况下它不再是抽象类而是具体类，它实现了在Abstraction中声明的抽象业务方法，在RefinedAbstraction中可以调用在Implementor中定义的业务方法。

    3.Implementor（实现类接口）：定义实现类的接口，这个接口不一定要与Abstraction的接口完全一致，事实上这两个接口可以完全不同，一般而言，Implementor接口仅提供基本操作，而Abstraction定义的接口可能会做更多更复杂的操作。Implementor接口对这些基本操作进行了声明，而具体实现交给其子类。通过关联关系，在Abstraction中不仅拥有自己的方法，还可以调用到Implementor中定义的方法，使用关联关系来替代继承关系。

    4.ConcreteImplementor（具体实现类）：给出Implementor接口的具体实现，而在不同的ConcreteImplementor中提供基本操作的不同实现，在程序运行时，ConcreteImplementor对象将替换其父类对象，提供给抽象类具体的业务操作方法。

    优点
    • 分离抽象接口及其实现部分，取代多层继承方案，降低了系统的复杂性，是解决多维度变化问题的有效方案。
    • 实现细节对客户透明，可以对用户隐藏实现细节。
    • 桥接模式有良好的扩展性，由于抽象与实现分离，因此扩展能力强，可以独立扩展抽象部分和实现部分，在一定程度上解耦了抽象部分和实现部分。
    缺点
    • 桥接模式的引入会增加系统的理解与设计难度，由于聚合关系建立在抽象层，要求开发者针对抽象进行设计与编程。
    • 桥接模式要求正确识别出系统中两个独立变化的维度，因此其使用范围有一定的局限性，如，一个类存在两个独立变化的维度，使用桥接模式进行重构。
    适用场景
    • 如果一个系统需要在构件的抽象化角色和具体化角色之间增加更多的灵活性，避免在两个层次之间建立静态的继承关系，通过桥接模式可以使它们在抽象层建立一个关联关系。
    • 一个类存在两个独立变化的维度，且这两个维度都需要进行扩展。
    • 不希望使用继承的场景。
    • 重用性要求较高的场景。
    类图

    以买咖啡为例子，咖啡有不同的口味，有不同的杯型，咖啡的口味和杯型是可以独立变化的，所以可以使用桥接模式来实现。

    7.享元模式

    享元模式（Flyweight Pattern）是运用共享技术有效地支持大量细粒度的对象。它通过共享已经存在的对象来大幅度减少需要创建的对象数量，从而降低系统的内存、CPU使用率。享元模式以共享的方式高效地支持大量细粒度的对象。

    大粒度对象：指的是占用内存空间大的对象，如：图片、视频等。

    享元模式的主要角色如下。

    1.抽象享元角色（Flyweight）：给出一个抽象接口，以规定出所有具体享元角色需要实现的方法。

    2.具体享元角色（Concrete Flyweight）：实现抽象享元角色所规定的接口。

    3.非共享具体享元角色（Unshared Concrete Flyweight）：并不是所有的抽象享元角色都需要被共享。Flyweight接口使共享成为可能，但它并不强制共享。在Flyweight对象结构的某些层次，UnsharedConcreteFlyweight对象通常将ConcreteFlyweight对象作为子节点。

    4.享元工厂角色（Flyweight Factory）：本角色负责创建和管理享元角色。本角色必须保证享元对象可以被系统适当地共享。当一个客户端对象调用一个享元对象的时候，享元工厂角色会检查系统中是否已经有一个符合要求的享元对象。如果已经有了，享元工厂角色就应当提供这个已有的享元对象；如果系统中没有一个适当的享元对象的话，享元工厂角色就应当创建一个合适的享元对象。

    优点
    • 减少对象的创建，降低内存中对象的数量，降低系统的内存，提高效率。
    • 外部状态相对独立，不影响内部状态，从而使得享元对象能够在不同的环境被共享。
    缺点
    • 享元模式使得系统更加复杂，需要分离出内部状态和外部状态，这使得程序的逻辑复杂化。
    • 为了使对象可以共享，享元模式需要将享元对象的状态外部化，而读取外部状态使得运行时间变长。
    • 维护共享对象，需要额外的开销。
    适用场景
    • 系统中存在大量的相似的大粒度对象。
    • 细粒度对象的创建和销毁造成了很大的开销。
    • 细粒度对象的大部分状态都可以外部化，可以将这些外部状态传入对象中，就使用享元模式来实现内外部状态的分离。
    • 需要缓冲池的场景。
